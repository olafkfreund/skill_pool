# MDM deployment — pushing `managed-settings.json` to every endpoint

This guide is for IT admins who already have a customised
`enterprise/managed-settings.json` and need to land it on every developer
machine in the fleet, per MDM tool. The file is what Claude Code reads at
startup to pin skill-pool's registry URL, tenant slug, and the
permissions baseline. Once it's on disk at the OS-specific path with the
right ownership, Claude Code merges it with the user's
`~/.claude/settings.json` and the project's `.claude/settings.json` on
every invocation.

## Scope

You are deploying **one file**, `managed-settings.json`, to a known path
on every developer endpoint. That's the whole job. Per-OS targets:

| OS | Path | Mode | Owner |
|---|---|---|---|
| macOS | `/Library/Application Support/ClaudeCode/managed-settings.json` | `0644` | `root:wheel` |
| Linux | `/etc/claude-code/managed-settings.json` | `0644` | `root:root` |
| Windows | `C:\ProgramData\ClaudeCode\managed-settings.json` | ACL: `SYSTEM`+`Administrators` full, `Users` read | `BUILTIN\Administrators` |

> The Windows location is `ProgramData` rather than `Program Files`
> because it's machine-wide *configuration* (per Microsoft's directory
> guidance), not a binary install root. `Program Files` is writable only
> to admins anyway, but `ProgramData` is the documented spot for
> machine-scoped settings files.

The input you're pushing is the file produced by following
`docs/enterprise/managed-settings.md` — typically generated by the
registry endpoint `GET /v1/enterprise/managed-settings` for your tenant.
A reference template lives at `enterprise/managed-settings.example.json`.

Deleting the file is a valid uninstall: Claude Code silently falls back
to user + project settings. There's no daemon to restart, no service to
stop. Drop the file → settings take effect on the *next* `claude`
invocation.

## Pre-flight

Before any MDM job goes out:

- [ ] **You have the file.** Generate it via the admin CLI / registry
      endpoint per `docs/enterprise/managed-settings.md`. Validate it
      locally: `jq . managed-settings.json` must succeed.
- [ ] **You have a test cohort.** A scope of 5–10 developers who'll
      get the file first and confirm `claude` still launches. Power
      users, not the CEO.
- [ ] **You have a rollback plan.** Rollback = your MDM tool pushes a
      revision that *removes* the file (not pushes an empty file —
      remove it). Claude Code reverts to user-only settings instantly.
- [ ] **The token file is handled separately.** This doc covers
      `managed-settings.json` only. The per-tenant API token at
      `/etc/skill-pool/token` (Linux) / equivalent paths is its own
      payload; see `docs/enterprise.md` § "Token rotation".
- [ ] **JSON is valid** (see Common pitfalls — a syntax error breaks
      Claude Code silently).

## Jamf Pro (macOS)

Two delivery patterns are common. Use whichever your org already
operates.

### Pattern A — Configuration Profile with "Application & Custom Settings"

Best when you want declarative drift correction: Jamf will re-push the
profile if anything diverges. The custom settings payload writes a
plist; we wrap our JSON in a single-key plist whose value is the literal
JSON string, then a small launchd job materialises it to the real path.
That sounds gnarlier than it is; in practice most macOS shops use
Pattern B for *files*, so jump to it unless your security team
specifically wants the profile-managed-everything approach.

### Pattern B — "Files and Processes" policy with a script payload (recommended)

1. **Upload the file as a package or attach as a script payload.** Place
   the JSON content in a script that writes it to disk, or upload
   `managed-settings.json` as a package whose postinstall does the move.

2. **Postinstall script:**

   ```bash
   #!/bin/bash
   set -euo pipefail

   TARGET_DIR="/Library/Application Support/ClaudeCode"
   TARGET="$TARGET_DIR/managed-settings.json"

   mkdir -p "$TARGET_DIR"
   cp -f "/tmp/managed-settings.json" "$TARGET"

   # Jamf scripts run as root; default umask leaves files 0600. Force 0644.
   chmod 0644 "$TARGET"
   chown root:wheel "$TARGET"

   # Validate JSON before declaring success.
   if ! /usr/bin/plutil -lint "$TARGET" >/dev/null 2>&1 \
        && ! /usr/bin/python3 -c "import json,sys; json.load(open(sys.argv[1]))" "$TARGET"; then
     echo "managed-settings.json failed JSON validation; reverting"
     rm -f "$TARGET"
     exit 1
   fi
   ```

3. **Scope the policy.**
   - Trigger: `Enrollment Complete` + `Recurring Check-in` (15 min
     default) for drift correction.
   - Scope: start with your test smart group; expand by phase.
   - Frequency: `Once per day` is plenty — the file barely changes.

4. **Verify in Jamf inventory.** Add an Extension Attribute that
   `cat`s the file and returns its hash, so the Jamf console can show
   which machines hold which revision.

> macOS gotcha: `cp` preserves source perms. If the script's source
> file is mode `0600` (the default for files written inside `/tmp`
> by a non-elevated process), the target will inherit `0600` and
> Claude Code can't read it unless invoked as the file's owner. Always
> `chmod 0644` after the copy.

## Microsoft Intune (Windows + macOS)

Be honest about the platform split: **Intune handles Windows and macOS
via completely different mechanisms**. There's no one-policy-fits-both.
You'll build two configurations.

### Windows — Win32 app deployment

Intune's "Custom configuration profile" with OMA-URI doesn't natively
handle "write this arbitrary JSON file to disk" the way it handles
registry keys. The clean path is a Win32 app.

1. **Package the file.** Use the Microsoft Win32 Content Prep Tool to
   wrap an `install.ps1` and the JSON into an `.intunewin` package.

2. **`install.ps1`:**

   ```powershell
   $ErrorActionPreference = 'Stop'

   $TargetDir = "$env:ProgramData\ClaudeCode"
   $Target    = Join-Path $TargetDir 'managed-settings.json'
   $Source    = Join-Path $PSScriptRoot 'managed-settings.json'

   New-Item -ItemType Directory -Force -Path $TargetDir | Out-Null
   Copy-Item -Force $Source $Target

   # Validate JSON parses.
   try {
     Get-Content $Target -Raw | ConvertFrom-Json | Out-Null
   } catch {
     Remove-Item $Target -Force
     throw "managed-settings.json failed JSON validation: $($_.Exception.Message)"
   }

   # Lock down ACLs: SYSTEM + Administrators full control, Users read.
   $acl = Get-Acl $Target
   $acl.SetAccessRuleProtection($true, $false)
   $acl.Access | ForEach-Object { $acl.RemoveAccessRule($_) | Out-Null }
   $rules = @(
     New-Object System.Security.AccessControl.FileSystemAccessRule(
       'NT AUTHORITY\SYSTEM', 'FullControl', 'Allow'),
     New-Object System.Security.AccessControl.FileSystemAccessRule(
       'BUILTIN\Administrators', 'FullControl', 'Allow'),
     New-Object System.Security.AccessControl.FileSystemAccessRule(
       'BUILTIN\Users', 'Read', 'Allow')
   )
   $rules | ForEach-Object { $acl.AddAccessRule($_) }
   Set-Acl -Path $Target -AclObject $acl
   ```

3. **`uninstall.ps1`** (for the rollback path):

   ```powershell
   $Target = Join-Path $env:ProgramData 'ClaudeCode\managed-settings.json'
   if (Test-Path $Target) { Remove-Item -Force $Target }
   ```

4. **Detection rule (Intune app settings):**

   - Rule type: `File`
   - Path: `%PROGRAMDATA%\ClaudeCode`
   - File: `managed-settings.json`
   - Detection method: `File or folder exists`

5. **Assignment.** Required → your test AAD group → expand by phase.

### macOS — Custom Configuration Profile (shell script + plist combo)

Intune for macOS supports running shell scripts on enrolled devices.
Use the **Devices → macOS → Shell scripts** flow:

1. **Script:** identical to the Jamf postinstall above (copies into
   `/Library/Application Support/ClaudeCode/`, `chmod 0644`,
   `chown root:wheel`, JSON validation).
2. **Run as signed-in user:** `No` (we need root).
3. **Frequency:** `Every 1 day` for drift correction.
4. **Retries:** `3`.

The JSON content has to be embedded *inside* the shell script as a
heredoc, because Intune's shell-script payload is one file. Example:

```bash
#!/bin/bash
set -euo pipefail
TARGET="/Library/Application Support/ClaudeCode/managed-settings.json"
mkdir -p "$(dirname "$TARGET")"
cat > "$TARGET" <<'EOF'
{
  "env": {
    "SKILL_POOL_REGISTRY": "https://acme.skill-pool.example.com",
    "SKILL_POOL_TENANT": "acme",
    "SKILL_POOL_TOKEN_FILE": "/etc/skill-pool/token"
  },
  "permissions": {
    "allow": ["Bash(skill-pool *)", "Read", "Glob", "Grep"],
    "deny":  ["Bash(rm -rf /*)", "Bash(rm -rf /)"]
  }
}
EOF
chmod 0644 "$TARGET"
chown root:wheel "$TARGET"
/usr/bin/python3 -c "import json,sys; json.load(open(sys.argv[1]))" "$TARGET" \
  || { rm -f "$TARGET"; exit 1; }
```

> Yes, this means the JSON lives inside the script as a heredoc — that's
> how Intune's macOS scripting works. For Windows you ship the JSON as
> a file inside the Win32 package; for macOS you embed it. If you'd
> rather centralise: have your MDM job templates source the JSON from
> the registry's `GET /v1/enterprise/managed-settings` endpoint and
> inject it at build-time before pushing the script to Intune.

## VMware Workspace ONE (UEM)

Workspace ONE is the cross-platform middle ground: one profile, many
endpoints, *if* you set it up right.

1. **Resources → Files/Actions → Add.** Upload your
   `managed-settings.json` as a file resource. Name it
   `skill-pool-managed-settings`.

2. **Define the deployment manifest.** For each OS, set the destination
   path and post-action.

   - **macOS:**
     - Destination: `/Library/Application Support/ClaudeCode/managed-settings.json`
     - Owner: `root`, group: `wheel`, permissions: `644`
     - Post-action (Shell): `/usr/bin/python3 -c "import json; json.load(open('/Library/Application Support/ClaudeCode/managed-settings.json'))"`

   - **Windows:**
     - Destination: `%PROGRAMDATA%\ClaudeCode\managed-settings.json`
     - Post-action (PowerShell): `Get-Content "$env:ProgramData\ClaudeCode\managed-settings.json" -Raw | ConvertFrom-Json | Out-Null`

   - **Linux** (Workspace ONE supports Linux via the Hub agent):
     - Destination: `/etc/claude-code/managed-settings.json`
     - Owner: `root:root`, permissions: `0644`
     - Post-action: `jq . /etc/claude-code/managed-settings.json > /dev/null`

3. **Smart Group assignment.** Same staged pattern as Jamf/Intune —
   start with the test cohort, ramp up.

4. **Profile sync cadence.** Workspace ONE checks in on the Hub agent's
   schedule (default 4h on macOS, 8h on Windows). For faster rollout
   during incident response, push an "Sync Device" command from the
   console.

> Workspace ONE's "Files Manager" payload was historically macOS-only;
> the cross-platform Files/Actions resource type is what's documented
> above and is the supported way to do this on modern UEM consoles.

## Ansible (Linux fleets)

The open-source / Linux-fleet path. Wrap the file into a role and let
your existing inventory + scheduling do the rest.

### Role layout

```
roles/skill-pool-managed-settings/
├── tasks/main.yml
├── handlers/main.yml
├── vars/main.yml
└── files/managed-settings.json
```

### `vars/main.yml`

```yaml
skill_pool_managed_settings_path: /etc/claude-code/managed-settings.json
skill_pool_managed_settings_owner: root
skill_pool_managed_settings_group: root
skill_pool_managed_settings_mode: "0644"
# SELinux context for RHEL/CentOS/Rocky/Fedora. Ignored elsewhere.
skill_pool_managed_settings_secontext: etc_t
```

### `tasks/main.yml`

```yaml
---
- name: ensure /etc/claude-code exists
  ansible.builtin.file:
    path: "{{ skill_pool_managed_settings_path | dirname }}"
    state: directory
    owner: "{{ skill_pool_managed_settings_owner }}"
    group: "{{ skill_pool_managed_settings_group }}"
    mode: "0755"

- name: deploy managed-settings.json
  ansible.builtin.copy:
    src: managed-settings.json
    dest: "{{ skill_pool_managed_settings_path }}"
    owner: "{{ skill_pool_managed_settings_owner }}"
    group: "{{ skill_pool_managed_settings_group }}"
    mode: "{{ skill_pool_managed_settings_mode }}"
    validate: "jq . %s >/dev/null"
  notify: verify claude code reads settings

- name: set SELinux context (RHEL family only)
  ansible.builtin.sefcontext:
    target: "{{ skill_pool_managed_settings_path }}"
    setype: "{{ skill_pool_managed_settings_secontext }}"
    state: present
  when: ansible_os_family == "RedHat"
  notify: restore selinux contexts

- name: confirm JSON parses on the target
  ansible.builtin.command:
    cmd: "jq . {{ skill_pool_managed_settings_path }}"
  changed_when: false
```

### `handlers/main.yml`

```yaml
---
- name: restore selinux contexts
  ansible.builtin.command:
    cmd: "restorecon -v {{ skill_pool_managed_settings_path }}"
  when: ansible_os_family == "RedHat"

- name: verify claude code reads settings
  ansible.builtin.command:
    cmd: claude --version
  changed_when: false
  failed_when: false
  # `claude --version` is a smoke test that the binary launches at all.
  # It does NOT verify the file's content was consumed — that's what
  # the staged rollout's developer feedback is for.
```

### Inventory invocation

```bash
ansible-playbook -i prod.yml site.yml \
  --tags skill-pool-managed-settings \
  --limit dev-cohort
```

The `validate:` parameter on `ansible.builtin.copy` runs `jq` against
the *staging* path before the file is moved into place. If the JSON is
malformed, the task fails and the existing file (if any) is preserved.

## Verification across all four

Identical post-deploy recipe per host:

1. **File exists at the OS-specific path.**

   - macOS: `test -f "/Library/Application Support/ClaudeCode/managed-settings.json"`
   - Linux: `test -f /etc/claude-code/managed-settings.json`
   - Windows: `Test-Path "$env:ProgramData\ClaudeCode\managed-settings.json"`

2. **JSON parses.**

   - Unix: `jq . <path>` exits 0
   - Windows: `Get-Content <path> -Raw | ConvertFrom-Json` doesn't throw

3. **Claude Code consumes it.** The cheap check is `claude --version`
   (binary still launches). The real check is invoking a skill that
   only your tenant's managed library publishes — e.g.
   `claude` then ask for a skill defined behind your registry's
   `SKILL_POOL_REGISTRY` URL. If the registry is unreachable or the
   `SKILL_POOL_TENANT` is wrong, the skill won't resolve and you'll
   know within seconds.

4. **MDM-side reporting.** Each tool offers some form of "did the
   profile apply?" answer:
   - Jamf: Extension Attribute returning the file's sha256
   - Intune: detection rule + `IntuneManagementExtension.log`
   - Workspace ONE: profile installation status per device
   - Ansible: `--diff` + `--check` on the next run

## Rolling out + rollback

**Staged rollout.** The default cadence that has burned the fewest
fleets:

1. **Wave 0 — 1 host.** You, the IT engineer. End-to-end manual
   verification.
2. **Wave 1 — ~10 hosts.** Voluntary early adopters. Run for 24–48h.
3. **Wave 2 — ~100 hosts.** Broader cohort, single business unit. Run
   for one full workday before expanding.
4. **Wave 3 — full fleet.**

At every wave, watch the registry's audit log for `claude-code` API
calls picking up new `SKILL_POOL_TENANT` headers — that's positive
confirmation the file took effect.

**Rollback.** The MDM update is "publish a revision that does not
contain the file *and* delete the existing file." Cookbook:

- **Jamf:** flip the policy to a removal-script payload that
  `rm -f "/Library/Application Support/ClaudeCode/managed-settings.json"`,
  re-scope to the affected smart group, force Recurring Check-in.
- **Intune (Windows):** mark the Win32 app's assignment as
  "Uninstall" for the affected AAD group. The `uninstall.ps1` deletes
  the file.
- **Intune (macOS):** push a shell script that `rm -f` the file. Scope
  it to the rollback cohort.
- **Workspace ONE:** remove the resource assignment from the smart
  group; the Hub agent will tombstone the file on next sync. To
  expedite, push a "Run Script" command that deletes the file.
- **Ansible:** run with `-e managed_settings_state=absent` and a
  conditional task:
  ```yaml
  - name: remove managed-settings.json (rollback)
    ansible.builtin.file:
      path: "{{ skill_pool_managed_settings_path }}"
      state: absent
    when: managed_settings_state | default('present') == 'absent'
  ```

After the file is gone, Claude Code resumes reading user + project
settings only. No restart needed.

## Common pitfalls

- **JSON syntax errors break Claude Code silently.** A trailing
  comma, a stray BOM, an unescaped backslash in a Windows path — the
  file fails to parse and Claude Code behaves as if no managed
  settings exist. *Always* validate with `jq .` (or PowerShell's
  `ConvertFrom-Json`) before pushing; every snippet in this guide
  includes the validation step for that reason.
- **macOS `cp` preserves source perms.** A copy from a `/tmp` file
  written by the running user lands as mode `0600`, root-owned but
  unreadable by the developer running `claude`. Always `chmod 0644`
  *after* the copy, never assume.
- **Windows ACL inheritance.** If you drop the file under
  `C:\ProgramData\ClaudeCode\` without explicit ACLs, it inherits
  whatever's set on `ProgramData` — usually fine, but Group Policy
  may have tightened it. The `Set-Acl` block in the PowerShell
  installer above sets explicit, non-inherited ACLs.
- **SELinux on RHEL/CentOS/Rocky/Fedora.** Files written into
  `/etc/claude-code/` by Ansible without `restorecon` may end up
  with an unexpected `unconfined_u:object_r:default_t:s0` context.
  Claude Code typically runs in the user's domain and can read it
  either way, but if you've hardened sysctl/SELinux beyond the
  defaults, the `sefcontext` + `restorecon` handler in the Ansible
  role keeps things sane. If you're on `setenforce 0`, ignore.
- **Stale token + new managed-settings.** Pushing a managed-settings
  that points at `SKILL_POOL_TENANT=acme` while the on-disk
  `/etc/skill-pool/token` still belongs to `acme-staging` will give
  developers 401s. Sequence rotations: token first, settings second.
- **Heredoc embedding on Intune macOS.** Once your JSON contains
  single quotes, the `'EOF'` quoted-heredoc form below still works,
  but bash variable expansion inside the JSON does not. If your
  template uses bash variables, drop the quotes around `EOF` and
  escape `$` characters in the JSON as `\$`.
